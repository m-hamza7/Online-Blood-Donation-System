Ch 2

STLC, Bug Life Cycle & Testing Fundamentals

Slide images — Chapter 2

Testing vs Debugging

Testing

  • Done by testers
  • Finds failures (observable wrong behaviour)
  • Reveals defects exist

Debugging

  • Done by developers
  • Finds, analyses & removes defects
  • Fixes root cause in code
Error → Defect → Failure: Human makes an error → creates a defect in code → execution causes a failure (visible symptom).

Testing Levels — The Pyramid

[UAT] Test basis: user stories Defects: business flow mismatches [System] Test basis: SRS Defects: functional failures [Integration] Test basis: architecture Defects: interface mismatches [Component/Unit] Test basis: code/data model Defects: logic errors

Component Testing

  • Individual modules in isolation
  • Done by developers (TDD)
  • Stubs & drivers used

Integration Testing

  • Module interactions
  • Big Bang / Top-Down / Bottom-Up
  • Interface defects

System Testing

  • End-to-end behaviour
  • Functional + Non-functional
  • Independent test team

UAT

  • User validates against business needs
  • Alpha (internal) / Beta (external)
  • Final acceptance gate
Each level has its own test basis, objectives, and typical defect types — know all three for each level.

Testing Types

Functional (WHAT it does)

  • Smoke — critical path gate before full testing
  • Sanity — quick check after a small fix
  • Regression — ensure new changes don't break old features

Non-Functional (HOW it performs)

  • Performance — load, stress, endurance
  • Usability — ease of use
  • Security — vulnerabilities
  • Reliability, Portability, Maintainability
Testing order: Smoke → Functional/Non-Functional → Regression

Verification vs Validation

Verification — “Building it RIGHT?”

  • Static — reviews, walkthroughs, inspections
  • Checks process and documents

Validation — “Building the RIGHT thing?”

  • Dynamic — actual execution
  • Checks final product vs user needs
A product can PASS verification but FAIL validation (built to spec, but wrong spec!).

V-Model

Requirements <--- test basis ---> Acceptance Testing (UAT) System Design <--- test basis ---> System Testing HLD <--- test basis ---> Integration Testing LLD <--- test basis ---> Unit/Component Testing [Coding]
Key principle: Each design phase on the left maps to a test phase on the right. Test planning starts at design.

Waterfall Model

  • Sequential phases: Requirements → Design → Implementation → Testing → Deployment
  • Testing only after FULL development
  • High fix cost (defects found late)
  • No late changes accepted
Waterfall: testing is last. Most expensive defects are found too late.

Severity vs Priority

Severity (Technical Impact)

  • Set by tester/QA
  • Critical / High / Medium / Low
  • App crashes = Critical severity

Priority (Business Urgency)

  • Set by stakeholders/PM
  • P1 / P2 / P3 / P4
  • CEO demo tomorrow = P1
Company logo wrong colour: Low severity (cosmetic) but High priority (board presentation today).
App crashes in rarely-used admin panel: High severity but Low priority (2 users affected).

Defect Metrics

DLR (Defect Leakage Rate) = (Defects missed / Total defects) x 100
DRR (Defect Removal Rate) = (Invalid reports / Total reported) x 100
Team found 39, missed 18: DLR = (18/58)x100 = 31.2%
Team raised 21, 5 invalid: DRR = (5/21)x100 = 23.8%

Agile — SCRUM & KANBAN

Agile Overview

Agile: Iterative-Incremental delivery. Frameworks: SCRUM, KANBAN, XP.

Advantages

  • Customer satisfaction via early delivery
  • Welcomes late requirement changes
  • Face-to-face daily communication
  • Rapid, frequent releases
  • Close developer-stakeholder cooperation

Disadvantages

  • Hard to estimate at project start
  • Poor documentation
  • Requires experienced team
  • Scope creep risk
  • Constant customer availability required

Scrum MasterFacilitator, removes impediments. NOT the project manager.
Development TeamCross-functional, self-organising. 3-9 people.

Static Testing

Slide images — Chapter 3

What is Static Testing?

Static Testing = examining work products WITHOUT executing the software.

Work Products Reviewed

  • SRS, user stories, use cases
  • Architecture & design docs
  • Source code
  • Test plans, test cases
  • Web pages, contracts

Issues Found

  • Requirement defects (ambiguous, missing)
  • Design defects (high coupling, low cohesion)
  • Coding defects (undefined vars, dead code)
  • Standards violations, security vulns
Static testing finds defects that dynamic testing CANNOT (e.g., requirement ambiguities before code is written).

False Positive vs False Negative

False Positive

  • Test FAILS but no real bug exists
  • Annoying but not dangerous
  • Wastes investigation time

False Negative

  • Test PASSES but a bug EXISTS
  • MORE DANGEROUS
  • Bug ships to production undetected

By Type

  • Compliance Audit — follows standards?
  • System Audit — whole QMS review
  • Process Audit — specific process
  • Product Audit — specific product

Dynamic Testing Techniques

Slide images — Chapter 4

Experience-Based Techniques

Error Guessing

  • Divide by zero
  • Null / empty inputs
  • Negative numbers
  • Max/min limits
  • Special characters (!@#$)
  • Very long strings

Exploratory Testing

  • Simultaneous learning, design, execution
  • Test charter — 1-2 hour time-boxed session
  • No pre-written scripts
  • Good for new/unknown features

White Box Coverage

Statement Coverage = every executable statement executed at least once.
Branch Coverage = every branch (true/false) executed. Subsumes statement coverage.
Condition Coverage = each atomic condition is both true and false.
TechniqueTest CaseInput
Statement CoverageTC01P=51, Q=01
Branch (true branch)TC01P=52, Q=01
Branch (false branch)TC02P=01, Q=01
Condition CoverageTC01P=00, Q=00
Branch Coverage is STRONGER than Statement Coverage. Always prefer branch when asked.

Boundary Value Analysis (BVA)

BVA: Test at the boundary, boundary-1, and boundary+1.
Age field: valid range 18-35
Valid boundaries: 18, 19, 34, 35
Invalid boundaries: 17 (just below), 36 (just above)
Order numbers 10000-99999: test values are 9999, 10000, 50000, 99999, 100000. Typical answer = 10000, 50000, 99999.

Tax Bands Exercise

BandRangeRate
A0-40000%
B4001-550010%

Tax boundary: 33501 is the BVA value for the 40% band entry.

Equivalence Partitioning (EP)

EP: Divide input into classes where all values behave the same. Test one from each class.
Age field 18-35:
Class 1 (Invalid): age < 18 — test: 10
Class 2 (Valid): 18-35 — test: 25
Class 3 (Invalid): age > 35 — test: 50
EP = 3 test cases (one per partition). BVA = 6 test cases (at each boundary). Combine for max coverage.

Decision Table Testing

Conditions TC1 TC2 TC3 TC4 Age > 18 Y Y N N Job Holder Y N Y N ------------------------------------------- Action: Approve X Action: Reject X X X
N conditions = 2^N maximum columns. Decision table tests all combinations.

Use Case Testing

Main Success Scenario (ATM): Insert card -> Validate card -> Enter PIN -> Grant access Extensions (alternative flows): 2a. Invalid card -> Eject card 3a. Wrong PIN (1st/2nd) -> Prompt retry 3b. Wrong PIN (3rd time) -> Retain/eat card
Use case tests cover the main flow PLUS all extensions.

State Transition Testing

State | Insert Card | Enter PIN | Wrong PIN | Correct PIN | Exit ----------|-------------|-----------|------------|-------------|----- S1 Idle | S2 | | | | S2 Card | | S3 | | | S1 S3 PIN | | | S4 / S7x3 | S5 Access | S4 Retry | | | S4 / S7x3 | S5 Access | S5 In Use | | | | | S1 S7 Locked | | | | |
Invalid transition = any event that leads to a state NOT defined in the table for that state/event combination.
Ch 5

Test Automation

Slide images — Chapter 5

Testing vs Test Automation

Manual Testing

  • WHAT to test
  • Test cases (scenarios)
  • Application domain expertise
  • Human judgment & creativity

Test Automation

  • HOW to test
  • Test scripts (code)
  • Development expertise
  • Speed & repeatability
Automation does NOT replace manual testing. It does NOT improve test effectiveness — only efficiency.

Automation Objectives & Anti-Patterns

Objectives

  • Improve testing efficiency
  • Wider coverage
  • Reduce long-term cost
  • Tests humans physically cannot do (load, concurrent)
  • Shorten test cycles (CI/CD)

Anti-Patterns (Avoid)

  • Scripts sensitive to minor UI changes
  • Hardcoded data values in scripts
  • OS-context-sensitive tests

ROI & Break-Even

Break-Even = Automation Cost / (Manual Cost Per Run - Automation Cost Per Run)
Automation costs 10,000. Manual run = 500, Automation run = 100.
Break-even = 10000 / (500 - 100) = 25 runs

Framework Architecture (Pg 23-25)

+--------------------------------------------------+ | TEST LAYER | | Test scripts, test suites, test runner | +--------------------------------------------------+ | +--------------------------------------------------+ | PAGE OBJECT LAYER | | Encapsulates page elements and actions | +--------------------------------------------------+ | +--------------------+ +------------------------+ | TEST DATA LAYER | | UTILITY LAYER | | JSON/CSV/Excel | | Helpers, wait utils | +--------------------+ +------------------------+ | +--------------------------------------------------+ | REPORTING LAYER | | HTML reports, screenshots, logs | +--------------------------------------------------+
Know all 5 layers: Test → Page Object → Test Data → Utility → Reporting.

Framework Evolution

Automation Framework = reusable set of libraries and classes. Reduces development AND maintenance cost.

SQA, Quality Tools & Cost of Quality

Slide images — Chapter 6

SQA vs SQC

SQA — Software Quality Assurance

  • Process-focused
  • PREVENTION
  • Ensures right processes are followed
  • Audits, standards, training

SQC — Software Quality Control

  • Product-focused
  • DETECTION
  • Finds defects in the product
  • Testing, inspection, reviews
SQA = process (prevention). SQC = product (detection). Classic exam trap!

3 SQA Principles

1. Know what you're doingUnderstand the current process thoroughly
2. Know what you SHOULD be doingUnderstand the target/standard process
3. Know how to measure the differenceMetrics to bridge current vs target gap

Cost of Quality (CoQ)

Prevention CostTraining, process documentation, quality planning. Spent BEFORE defects occur.
Appraisal CostTesting, reviews, audits. Finding defects early.
Internal FailureRework, retesting. Discovered BEFORE delivery.
External FailureWarranty, recalls, reputation damage. MOST EXPENSIVE!
External failure is the most expensive. Prevention is the cheapest. Pay now or pay much more later.

Head = problem. Bones = cause categories. Sub-bones = specific causes. Used for root cause analysis.

Risk Management

Slide images — Chapter 11

Risk Fundamentals

Risk = probability of loss OR gain while pursuing goals due to unpredictable factors.

Internal Risk

From process/organisational inadequacies. Within your control.

Examples: poor planning, skill gaps, tool failures

External Risk

Uncontrollable external conditions.

Examples: new regulations, natural disasters, market shifts

Risk Terms

TermDefinition
Risk IDUnique identifier for the risk
ProbabilityLikelihood of occurrence (0-1 or %)
ImpactConsequence if it occurs (1-5 scale)
ExposureProbability x Impact
OriginSource/cause of the risk
CategoryType (technical, organisational, etc.)
OwnerPerson responsible for managing the risk
Risk Exposure = Probability x Impact

Risk Management Process

1. Identification
→
2. Analysis & Assessment
→
3. Response Plan
→
4. Monitoring & Control

Risk Identification Methods

Type I — Intuitive

  • Mind Mapping — visual risk brainstorm
  • Brainstorming — 6 steps: Invite, Explain, Time Limit, Encourage, Filter, Finalise
  • Out-of-Box Thinking

Type I — Historical

  • Caper Jones Top 10: Creeping requirements, cancelled projects, excessive schedule pressure, inadequate design...
  • Rex Black Top 10: Functionality, load, reliability, security, compatibility, usability...

Risk Screening Categories

HazardMurphy's Law: if it CAN go wrong it WILL. Need early warning systems.
Constraint100% certain to occur. Project always runs within these.
NormalPareto: 20% of risks cause 80% of problems.
TrivialLow significance. Keep aside, monitor occasionally.

Risk Quadrant Map

HIGH IMPACT | Q2 | Q1 High Impact | HIGH IMPACT Low Prob | HIGH PROB (MONITOR) | (TOP PRIORITY!) | LOW PROB ---------+--------- HIGH PROB | Q4 | Q3 Low Impact | Low Impact Low Prob | High Prob (ACCEPT/IGNORE) | (MONITOR) | LOW IMPACT
Q1 (High Impact + High Prob) = must address immediately. Q4 (Low+Low) = accept and ignore.

Risk Responses — Negative Risks (Threats)

StrategyActionShark Example
AvoidChange strategy, remove root causeSwim in a shark cage
MitigateReduce probability or impactUse shark bait to distract
TransferChange risk ownerHire professional shark diver
Accept (Active)Prepare contingency planCarry a spear gun as Plan B
Accept (Passive)Acknowledge and do nothingSwim and hope for the best
ContingencyPlan B triggered when risk occursEmergency evacuation plan

Risk Responses — Positive Risks (Opportunities)

StrategyActionBonus Example
ExploitGuarantee the opportunity happensAssign senior programmer to deliver early
EnhanceIncrease probability of opportunityTrain programmer to speed up delivery
ShareDistribute opportunity to partnersContract out to specialist team
AcceptTake advantage if it happens; do nothing specialDo nothing, hope bonus comes
Negative: Avoid, Mitigate, Transfer, Accept, Contingency. Positive: Exploit, Enhance, Share, Accept. "Accept" applies to BOTH!

Causal Analysis

Risk: Project Delay |-- Cause: Scope Creep |-- Sub-Cause: Unclear requirements |-- Root Cause: No formal change control process
Causal analysis: Risk → Cause → Sub-Cause → Root Cause. Fix the root cause to prevent the risk recurring.

What is MCP?

MCP (Model Context Protocol) = standard rules allowing an LLM to connect to external tools and ACT — not just generate text.

LLM Alone

  • Can write code, queries, essays
  • CANNOT execute anything
  • No file system access
  • Text only

LLM + MCP

  • Can execute tests
  • Can debug failures
  • Can self-heal broken tests
  • Full tool integration

Playwright MCP Setup

  • Uses accessibility tree — NOT screenshots
  • LLM-friendly structured representation
  • Deterministic & reliable locators
